# NovaNet L4 Load Balancer Design

> eBPF-based kube-proxy replacement for Kubernetes Service DNAT

## Overview

NovaNet L4 LB replaces kube-proxy entirely with eBPF-based Service DNAT. It handles all Kubernetes Service types (ClusterIP, NodePort, ExternalIP, LoadBalancer) directly in TC hooks, with connection tracking via eBPF LRU hash maps. When NovaEdge is installed, it supersedes NovaNet's L4 LB with full L7 capabilities.

## Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Service types | All (ClusterIP, NodePort, ExternalIP, LoadBalancer) | Full kube-proxy replacement from day one |
| Backend selection | Random, Round-Robin, and Maglev — per service | Different workloads benefit from different strategies |
| Connection tracking | eBPF LRU hash map | Self-contained, no kernel version deps, proven (Cilium pattern) |
| Rollout strategy | Hard replace | Disable kube-proxy entirely when L4 LB enabled |

## Architecture

### Components

1. **Go agent** (`internal/service/`) — Service/EndpointSlice watcher, pushes to dataplane via gRPC
2. **Rust dataplane** — new gRPC RPCs to populate eBPF maps
3. **eBPF programs** — DNAT in `tc_egress`, reverse SNAT in `tc_ingress`/`tc_tunnel_ingress`, NodePort DNAT on host interface
4. **eBPF maps** — 4 new: SERVICES, BACKENDS, CONNTRACK, MAGLEV

### Data Flow

**ClusterIP (pod → service):**
```
Pod sends to 10.43.100.73:8080 (ClusterIP)
  → tc_egress on pod veth
  → check CONNTRACK → if hit, apply cached DNAT
  → check SERVICES map: (dst_ip, dst_port, proto, scope=ClusterIP)
  → select backend (random/round-robin/maglev)
  → create CONNTRACK entry: (src, dst_vip) → (backend_ip, backend_port)
  → rewrite dst IP/port (DNAT)
  → incremental checksum update (bpf_l3_csum_replace + bpf_l4_csum_replace)
  → continue to existing logic (same-node or tunnel encap)
```

**Return path (backend → pod):**
```
Backend reply arrives at pod's veth
  → tc_ingress (or tc_tunnel_ingress for cross-node)
  → check CONNTRACK by reverse tuple
  → rewrite src IP/port back to Service VIP/port (reverse SNAT)
  → incremental checksum update
  → deliver to pod
```

**NodePort (external → service):**
```
External client → bond0:30080
  → tc_host_ingress on physical interface
  → lookup SERVICES(0, 30080, TCP, scope=NodePort)
  → DNAT to backend pod IP/port
  → route to pod (local or remote via tunnel)
```

After DNAT, the dst_ip is a real pod IP, so existing endpoint lookup and tunnel/routing logic works unchanged.

## eBPF Maps

### SERVICES — `HashMap<ServiceKey, ServiceValue>`

```rust
#[repr(C)]
struct ServiceKey {
    ip: u32,           // VIP (ClusterIP/ExternalIP, or 0 for NodePort)
    port: u16,         // Service port
    protocol: u8,      // TCP=6, UDP=17, SCTP=132
    scope: u8,         // 0=ClusterIP, 1=NodePort, 2=ExternalIP, 3=LoadBalancer
}

#[repr(C)]
struct ServiceValue {
    backend_count: u16,    // Number of backends
    backend_offset: u16,   // Starting index in BACKENDS array
    algorithm: u8,         // 0=random, 1=round-robin, 2=maglev
    flags: u8,             // Bit 0: session affinity, Bit 1: externalTrafficPolicy=Local
    affinity_timeout: u16, // Session affinity timeout seconds (0 = disabled)
    maglev_offset: u32,    // Offset into MAGLEV lookup table (if algorithm=2)
}
```

Max entries: 16384

### BACKENDS — `Array<BackendValue>`

```rust
#[repr(C)]
struct BackendValue {
    ip: u32,           // Backend pod IP
    port: u16,         // Backend target port
    node_ip: u32,      // Node hosting backend (for externalTrafficPolicy: Local)
    _pad: [u8; 2],
}
```

Max entries: 65536 (flat array, services index into it by offset)

### CONNTRACK — `LruHashMap<CtKey, CtValue>`

```rust
#[repr(C)]
struct CtKey {
    src_ip: u32,
    dst_ip: u32,       // Original destination (the VIP)
    src_port: u16,
    dst_port: u16,
    protocol: u8,
    _pad: [u8; 3],
}

#[repr(C)]
struct CtValue {
    backend_ip: u32,    // DNAT'd destination
    backend_port: u16,  // DNAT'd port
    origin_ip: u32,     // Original VIP (for reverse SNAT)
    origin_port: u16,   // Original service port
    flags: u8,          // TCP state tracking bits
    _pad: u8,
    timestamp: u64,     // Last seen (for affinity)
}
```

Max entries: 524288 (512K, LRU eviction handles cleanup)

### MAGLEV — `Array<u32>`

Flat lookup table. Each service with `algorithm=maglev` gets a 65537-entry block. The value at each index is the backend index (relative to the service's `backend_offset`). Maglev tables are generated in userspace (Go agent) and pushed to the map.

Max entries: 1048576 (supports ~16 Maglev services)

## Go Agent: Service Watcher

### Package: `internal/service/`

**Watchers:**
- `v1.Service` — on add/update/delete, compute service map entries (one per port × scope)
- `discovery.k8s.io/v1.EndpointSlice` — on change, recompute backend list for owning service

**Reconciliation logic:**
1. Service change or EndpointSlice change triggers reconcile
2. Build full backend list (all ready endpoints across all EndpointSlices for the service)
3. Allocate contiguous slot in backend array (reuse or allocate new range)
4. For Maglev services, regenerate lookup table in userspace
5. Push via gRPC: `UpsertService` + `UpsertBackends`

**Backend slot allocator:**
Simple allocator tracking used/free ranges in the flat backend array. Reuses same offset when backend count unchanged, allocates new range + frees old when count changes.

**Service type → map entries:**

| Service Type | Entries Created |
|-------------|----------------|
| ClusterIP | 1 entry per port: key=(clusterIP, port, proto, scope=0) |
| NodePort | ClusterIP entry + 1 per port: key=(0, nodePort, proto, scope=1) |
| ExternalIP | ClusterIP entry + 1 per externalIP per port: key=(extIP, port, proto, scope=2) |
| LoadBalancer | ClusterIP + NodePort entries + 1 per LB ingress IP: key=(lbIP, port, proto, scope=3) |


## Configuration

New config field: `l4lb_enabled: bool` (default: `false`)

When enabled:
- Go agent starts Service/EndpointSlice watchers
- Go agent attaches TC programs to host physical interface (for NodePort/ExternalIP)
- Sets `CONFIG_KEY_L4LB_ENABLED` in eBPF config map
- eBPF programs perform service lookups before endpoint lookups

**Deployment requirement:** k3s must be started with `--disable-kube-proxy` when L4 LB is enabled.

## eBPF Program Changes

### tc_egress (pod veth) — DNAT

Insert before existing endpoint lookup:
1. Check CONNTRACK → if hit, apply cached DNAT, skip backend selection
2. Check SERVICES map with (dst_ip, dst_port, protocol, scope=ClusterIP)
3. If service found: select backend, create CT entry, rewrite headers, recalculate checksums
4. If no match: fall through to existing behavior

### tc_ingress (pod veth) — Reverse SNAT

On return traffic:
1. Check CONNTRACK by reverse tuple (src=backend, dst=pod)
2. If hit: rewrite src IP/port back to VIP/port, recalculate checksums
3. Continue to existing logic

### tc_tunnel_ingress — Cross-node reverse SNAT

Same reverse SNAT logic for traffic arriving via Geneve/VXLAN tunnel.

### NEW: tc_host_ingress / tc_host_egress

Attached to node's physical interface (bond0). Handles NodePort and ExternalIP DNAT for external traffic entering the node.

### Checksum Updates

Use `bpf_l3_csum_replace` (IP header) and `bpf_l4_csum_replace` (TCP/UDP) for incremental checksum updates. Only changed fields (IPs, ports) are patched.

## Backend Selection Algorithms

### Random (algorithm=0)
Hash 5-tuple with `bpf_get_prandom_u32()`, mod `backend_count`. Simple, O(1), good distribution under load.

### Round-Robin (algorithm=1)
Per-service atomic counter stored in a `PerCpuArray` map. Increment and mod `backend_count`. Even distribution at low connection rates.

### Maglev (algorithm=2)
Lookup `MAGLEV[maglev_offset + hash(5-tuple) % 65537]` → backend index. Consistent hashing ensures minimal disruption when backends change. Table generated in Go userspace and pushed to eBPF.
